# software-quality-handbook
CA assignment for Software Quality Assurance 

## Overview of project 

This is a software quality handbook made for a small fast paced startup engineering team, this is to solve the issue where individual engineers are following inconsistent practices that result in unpredictable progress and defects reaching production.

## Purpose of the handbook 

The purpose of this handbook is to create a clear and practical standard for software engineers for how they should be approaching software quality practices, our design allows them to be precise, practical and we made it very easy and user friendly.

Software engineers in a company especially a start up tend to develop different ways in which they estimate their work, review code and maintain quality throughout, due to each one using their own version this leads to inconsistent delivery and poor collaboration amongst collegues that can lead to very avoidable bugs. 

Here we establosh what the "right way" of working should be in the following areas:
- Task Estimation in Scrum
- Code reviews
- Testing strategies

Our research outlines what the best practices to follow are, how to avoid bad practices and what some common themes are across these topics.

## Goals

Our main goals for this project is that we:

- Create consistent and practical software quality practices for a small engineering startup team.
- Support onboarding process for new engineers.
- Reduce defects from reaching production.
- Improve the consistency and predictablity in delivery.
- Show how effective teamwork can be through the process of planning, pull requests, and sharing contributions.

We used a trunk based devlopment workflow and we achieved this by completing each major task on seperate feature branches and merging them into the mainn through PRs, we have both contributed to the commits to the repo and reviewed pull requests and gave feedback. We have also included retrospectives after completing our sections to see what we could do differently in the future.
