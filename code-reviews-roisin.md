#Code Reviews

##Introduction
Code reviews are an important way for small startup teams to keep software quality high. These reviews are needed in order to help engineers catch bugs or problems before code goes live. 
It's easier to maintain and read the code, and make sure everyone follows the same standards. In a fast-paced team, code reviews also help everyone stay on the same page, which prevents unnecessary bugs and ensures consistent quality.
 
A good code review is more than just spotting mistakes, it helps everyone share knowledge, work better as a team and keep the standards consistent. For new team members, 
regular reviews make it easier to learn how they write code, give feedback, and define quality.

##Common Themes
Several common themes appear again and again within code reviews:

- Keep pull requests small enough to review properly.
- Review code soon after it is opened.
- Focus feedback on the code, not the person who wrote it.
- Check whether the change matches the goal described in the pull request.
- Look for readability, maintainability, and possible defects, not just whether the code works right now.
- Use code reviews to share knowledge across the team.
- Be respectful, specific, and constructive when giving feedback.
- Make sure the review process is consistent so all engineers know what is expected.

These themes matter especially in a startup because speed can easily lead to rushed reviews, vague comments, or large pull requests that nobody wants to read carefully.

##Revieweer Expectations
A reviewer should approach a pull request to improve the quality of the code and help the team maintain a shared standard.

Reviewers should:
- Read the pull request title and description first.
- Understand the purpose of the change before reviewing the code itself.
- Check whether the code solves the problem it claims to solve.
- Look for bugs, confusing logic, duplication, and unclear naming.
- Check whether the code is readable and easy for another engineer to maintain.
- Consider whether tests are included where needed.
- Give comments that are specific and helpful.
- Keep feedback professional and respectful.
- Approve changes once they meet the expected standard.
- Avoid delaying reviews without reason.
