# Code Reviews

## Introduction

Code reviewing is one of the most effective ways to improve software quality in a team environment. It helps catch defects before code reaches production, improves consistency and allows for constructive feedback.

In a small startup poor code review habits can lead to rushed merges, hidden bugs, unclear ownership, and inconsistent standards.
Here we define how code reviews should be approached in this company so that they are efficient and support quality.

## Common themes

- Code reviews improve quality when they focus on important issues
- Good reviews are constructive, respectful, and specific
- Smaller pull requests are easier to review well
- Code review helps spread knowledge and maintain team standards

## Best practices

- Review accuracy, readability, maintainability.
- Focus on the most important risks first
- Give feedback that is clear, respectful, and specific
- Explain why a change is needed when leaving comments
- Ask questions when something is unclear rather than making assumptions
- Review code in a reasonable timeframe
- Encourage consistency without turning every review into a debate over personal style

## Bad practices

- Leaving vague comments with no explanation
- Taking feedback personally
- Opening very large pull requests that are hard to review properly
- Treating code review as a formality instead of a quality step

## Team rules for the company

- Every pull request should be reviewed by at least one other engineer before merging
- Pull requests should include enough context for reviewers to understand the change
- Reviews should focus on quality, clarity, and maintainability
- Feedback should stay professional and constructive
